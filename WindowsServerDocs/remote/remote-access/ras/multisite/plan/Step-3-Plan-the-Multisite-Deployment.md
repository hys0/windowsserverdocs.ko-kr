# <a name="step-3-plan-the-multisite-deployment"></a>3 단계 계획 멀티 사이트 배포

>적용 대상: Windows Server (반기 채널), Windows Server 2016

멀티 사이트 인프라를 계획 한 후 추가 인증서 요구 사항을 계획 클라이언트 컴퓨터 항목 지점 및 배포에 할당 된 IPv6 주소를 선택 하는 방법입니다.  

다음 섹션에서는 자세한 계획 정보를 제공 합니다.
  
## <a name="bkmk_3_1_IPHTTPS"></a>3.1 계획 IP-HTTPS 인증서  
진입점을 구성 하면 특정 ConnectTo 주소를 사용 하 여 각 진입점을 구성 합니다. 각 진입점에 대 한 IP-HTTPS 인증서 ConnectTo 주소를 일치 해야 합니다. 인증서를 가져올 때 다음 note:  
  
-   멀티 사이트 배포에서는 자체 서명된 인증서를 사용할 수 없습니다.  
  
-   CRL을 항상 사용할 수 있도록 공용 CA를 사용하는 것이 좋습니다.  
  
-   제목 필드 지정 (ConnectTo 주소는 IP 주소 및 DNS 이름이 아닌으로 지정) 하는 경우 원격 액세스 서버의 외부 어댑터의 IPv4 주소 또는 IP-HTTPS URL의 FQDN입니다.  
  
-   인증서의 일반 이름 IP-HTTPS 웹 사이트의 이름과 일치 해야 합니다. ConnectTo DNS 이름과 일치 하는 와일드 카드 URL의 사용도 지원 됩니다.  
  
-   IP-HTTPS 인증서 주체 이름에 와일드 카드를 사용할 수 있습니다. 모든 진입점에 대 한 동일한 와일드 카드 인증서를 사용할 수 있습니다.  
  
-   확장된 키 사용 필드의 경우 서버 인증 OID(개체 식별자)를 사용합니다.  
  
-   멀티 사이트 배포에서 Windows 7을 실행 하는 클라이언트 컴퓨터를 지 원하는 경우 CRL 배포 지점 필드의 경우에서 인터넷에 연결 된 DirectAccess 클라이언트에서 액세스할 수 있는 CRL 배포 지점을 지정 합니다. 이 기본적으로 CRL 해지 확인을 사용할 수 없게 됩니다 IP-HTTPS이이 클라이언트에 Windows 8을 실행 하는 클라이언트에 대 한 필요 하지 않습니다.  
  
-   IP-HTTPS 인증서에 프라이빗 키가 있어야 합니다.  
  
-   IP-HTTPS 인증서, 컴퓨터 및 사용자의 개인 저장소로 직접 가져와야 합니다.  
  
## <a name="bkmk_3_2_NLS"></a>3.2 네트워크 위치 서버 계획  
조직에서 다른 서버 또는 원격 액세스 서버의 네트워크 위치 서버 웹 사이트를 호스트할 수 있습니다. 원격 액세스 서버의 네트워크 위치 서버를 호스트 하는 경우 원격 액세스를 배포 하는 경우 웹 사이트를 자동으로 만들어집니다. 조직에서 Windows 운영 체제를 실행 하는 다른 서버의 네트워크 위치 서버를 호스트 하는 경우 인터넷 정보 서비스 (IIS) 웹 사이트를 설치 되어 있는지 확인 해야 합니다.  
  
### <a name="321-certificate-requirements-for-the-network-location-server"></a>3.2.1 네트워크 위치 서버에 대 한 요구 사항 인증서  
네트워크 위치 서버 웹 사이트 인증서 배포에 대 한 다음 요구 사항을 충족 하는지 확인 합니다.  
  
-   HTTPS 서버 인증서가 필요합니다.  
  
-   네트워크 위치 서버가 원격 액세스 서버에 있는 단일 원격 액세스 서버를 배포할 때 자체 서명 된 인증서를 사용 하도록 선택한 경우에 내부 CA에서 발급 한 인증서를 사용 하 여 단일 서버 배포를 다시 구성 해야 합니다.  
  
-   DirectAccess 클라이언트 컴퓨터가 네트워크 위치 서버 웹 사이트에 서버 인증서를 발급한 CA를 신뢰해야 합니다.  
  
-   내부 네트워크에 DirectAccess 클라이언트 컴퓨터는 네트워크 위치 서버 웹 사이트의 이름을 확인할 수 있어야 합니다.  
  
-   네트워크 위치 서버 웹 사이트는 내부 네트워크의 컴퓨터에 항상 사용할 수 있어야 합니다.  
  
-   인터넷의 DirectAccess 클라이언트 컴퓨터에서 네트워크 위치 서버에 액세스할 수 없어야 합니다.  
  
-   인증서 해지 목록 (CRL)에 대 한 서버 인증서를 선택 해야 합니다.  
  
-   네트워크 위치 서버가 원격 액세스 서버에서 호스팅되는 경우에 와일드 카드 인증서 지원 되지 않습니다.  
  
네트워크 위치 서버에 사용할 웹 사이트 인증서를 가져온 경우 다음 note:  
  
1.  주체 필드에 네트워크 위치 서버의 인트라넷 인터페이스 IP 주소 또는 네트워크 위치 URL의 FQDN을 지정합니다. 네트워크 위치 서버가 원격 액세스 서버에서 호스팅되는 경우 IP 주소를 지정 하지는 참고 합니다. 네트워크 위치 서버는 모든 진입점에 대 한 동일한 주체 이름을 사용 해야 하 고 동일한 IP 주소가 모든 진입점 때문입니다.  
  
2.  확장된 키 사용 필드의 경우 서버 인증 OID를 사용합니다.  
  
3.  CRL 배포 지점 필드의 경우 인트라넷에 연결 된 DirectAccess 클라이언트에서 액세스할 수 있는 CRL 배포 지점을 사용 합니다.  
  
### <a name="322dns-for-the-network-location-server"></a>네트워크 위치 서버에 대 한 3.2.2DNS  
원격 액세스 서버의 네트워크 위치 서버를 호스트 하는 경우 배포에서 모든 진입점에 대 한 네트워크 위치 서버 웹 사이트에 대 한 DNS 항목을 추가 해야 합니다. 다음에 유의하세요.  
  
-   따라서 주체 이름 및 네트워크 위치 서버 URL에 같을 수 없습니다 컴퓨터 이름으로, 모든 진입점에 대 한 멀티 사이트 배포의 첫 번째 네트워크 위치 서버 인증서의 주체 이름으로 네트워크 위치 서버 URL에 포함 되는 배포의 첫 번째 원격 액세스 서버입니다. 네트워크 위치 서버에 대 한 전용된 FQDN 이어야 합니다.  
  
-   네트워크 위치 서버 트래픽은에서 제공 하는 서비스는 DNS를 사용 하는 진입점 분산 및 진입점의 내부 IP 주소로 구성 된 각 진입점에 대 한 동일한 URL 사용 하 여 DNS 항목을 해야 하므로 합니다.  
  
-   모든 진입점 (일치 하는 네트워크 위치 서버 URL) 동일한 주체 이름 가진 네트워크 위치 서버 인증서를 사용 하 여 구성 되어야 합니다.  
  
-   네트워크 위치 서버 인프라 (DNS 및 인증서 설정)를 진입점에 대 한 진입점을 추가 하기 전에 만들어야 합니다.  
  
## <a name="bkmk_3_3_IPsec"></a>3.3 모든 원격 액세스 서버에 대 한 IPsec 루트 인증서 계획  
멀티 사이트 배포에서 IPsec 클라이언트 인증에 대 한 계획을 수립할 때 다음 note:  
  
1.  Kerberos 프록시를 멀티 사이트에 대 한 지원 되지 않으므로 내부 CA에서 발급 하는 컴퓨터 인증서를 사용 하도록 설정을 변경 해야 단일 원격 액세스 서버를 설정할 때 컴퓨터 인증을 위한 기본 제공 Kerberos 프록시를 사용 하려는 경우 배포 합니다.  
  
2.  자체 서명 된 인증서를 사용 하는 경우에 내부 CA에서 발급 한 인증서를 사용 하 여 단일 서버 배포를 다시 구성 해야 합니다.  
  
3.  클라이언트 인증 시 성공 하려면 IPsec 인증에 대 한 모든 원격 액세스 서버에 확장 된 키 사용에 대 한 클라이언트 인증 OID와 IPsec 루트 또는 중간 CA를 발급 한 인증서가 있어야 합니다.  
  
4.  동일한 IPsec 루트 또는 중간 인증서를 모든 멀티 사이트 배포에서 원격 액세스 서버에 설치 되어야 합니다.  
  
## <a name="bkmk_3_4_GSLB"></a>3.4 전역 서버 부하 분산 계획  
멀티 사이트 배포에서는 전역 서버 부하 분산 장치를 추가로 구성할 수 있습니다. 전역 서버 부하 분산 장치 배포에 진입점 간에 트래픽 부하를 배포할 수 있으므로 큰 지리적 배포를 포함 하는 경우 조직에 유용할 수 있습니다.  DirectAccess 클라이언트에 가장 가까운 진입점의 항목 지점 정보를 제공 하는 전역 서버 부하 분산 장치를 구성할 수 있습니다. 프로세스는 다음과 같이 작동합니다.  
  
1.  Windows 10 또는 Windows 8을 실행 하는 클라이언트 컴퓨터 전역 서버 부하 분산 장치 IP 주소 목록을 각 진입점과 연결 합니다.  
  
2.  Windows 10 또는 Windows 8 클라이언트 컴퓨터의 공용 DNS에서 전역 서버 부하 분산 장치 FQDN을 IP 주소로 확인 하려고 합니다. 확인 된 IP 주소는 진입점의 전역 서버 부하 분산 장치 IP 주소와 나열 된 경우 클라이언트 컴퓨터는 자동으로 해당 진입점 선택한 Teredo 서버 IP 주소 또는 IP-HTTPS URL (ConnectTo 주소)에 연결 합니다. 전역 서버 부하 분산 장치의 IP 주소를 동일 하 게 또는 Teredo 서버 주소는 진입점의 ConnectTo 주소를 클라이언트 컴퓨터에서 전역 서버 부하 분산 장치 IP 주소에 연결 하려고 하지도 말아야 하므로 필요 없는 참고 합니다.  
  
3.  전역 서버 부하 분산 장치 IP 주소를 구성 하는 모든 클라이언트 컴퓨터가 프록시 뒤에 있는 웹 (및 DNS 확인을 사용할 수 없습니다) 하거나 전역 서버 부하 분산 장치 FQDN을 해결 되지 않으면을 진입점에 HTTPS 프로브를 사용 하 여 자동으로 선택 됩니다. 모든 진입점에서 IP-HTTPS Url입니다. 클라이언트가 가장 먼저 응답 하는 서버에 연결 됩니다.  
  
전역 서버 부하 분산 원격 액세스를 지 원하는 장치 목록은,로 이동 찾기에 파트너 페이지 [Microsoft 서버 및 클라우드 플랫폼](https://www.microsoft.com/server-cloud/)합니다.  
  
## <a name="bkmk_3_5_EP_Selection"></a>3.5 계획 DirectAccess 클라이언트 항목 지점 선택  
배포의 모든 진입점에 연결 하 고 선택 항목을 단일 진입점을 자동으로 연결할 때 기본적으로 Windows 10을 멀티 사이트 배포를 구성 하 고 Windows 8 클라이언트 컴퓨터 정보를 사용 하 여 구성 됩니다 필요 알고리즘입니다. Windows 10을 허용 하도록 배포를 구성할 수도 있습니다 마우스 Windows 8 클라이언트 컴퓨터를 수동으로 항목을 선택 하 고 연결 됩니다. Windows 10 또는 Windows 8 클라이언트 컴퓨터는 현재 미국 진입점에 연결 하는 경우 자동 항목 지점 선택을 사용 하는 미국 진입점에 연결할 수 없는 경우 몇 분 후 클라이언트 컴퓨터 연결 하려고 합니다. 통해 유럽 진입점입니다. 자동 항목 지점 선택 영역을 사용 하 여 것이 좋습니다. 그러나 수동 입력 지점을 선택 하면 최종 사용자가 현재 네트워크 조건에 따라 다른 진입점에 연결할 수 있습니다. 예를 들어 컴퓨터는 미국 진입점 및 내부 네트워크에 대 한 연결에 연결 되어 있으면가 예상 보다 훨씬 느립니다. 이 경우 최종 사용자는 내부 네트워크에 대 한 연결을 개선 하기 위해 유럽 진입점에 연결할 수동으로 선택할 수 있습니다.  
  
> [!NOTE]  
> 최종 사용자는 진입점을 수동으로 선택, 후 클라이언트 컴퓨터 하지 지점 선택 항목이 자동으로 되돌아갑니다. 즉, 수동으로 선택한 진입점에 연결할 수 없는 경우 최종 사용자 자동 입력 지점 선택으로 되돌리려면 하거나 다른 진입점을 수동으로 선택 합니다.  
  
 Windows 7 클라이언트 컴퓨터는 단일 진입점을 멀티 사이트 배포에서에 연결 하는 데 필요한 정보를 사용 하 여 구성 됩니다. 동시에 여러 진입점에 대 한 정보를 저장할 수는 없습니다. 예를 들어 미국 진입점 하지만 하지 유럽 진입점에 연결 하는 Windows 7 클라이언트 컴퓨터를 구성할 수 있습니다. 미국 진입점에 연결할 수 없는 경우 Windows 7 클라이언트 컴퓨터는 진입점이 도달할 때까지 내부 네트워크에 대 한 연결을 손실 됩니다. 최종 사용자 유럽 진입점에 연결 하려고 변경할 수 없습니다.  
  
## <a name="bkmk_3_6_IPv6"></a>3.6 접두사 및 라우팅을 계획합니다  
  
### <a name="internal-ipv6-prefix"></a>내부 IPv6 접두사  
단일 원격 액세스 배포 중 내부 네트워크 IPv6 접두사를 계획 하는 서버에 멀티 사이트 배포에서 다음을 유의 하십시오.  
  
1.  단일 서버 원격 액세스 배포를 구성 하는 경우 Active Directory 사이트를 모두 포함할, 내부 네트워크 IPv6 접두사는 이미 원격 액세스 관리 콘솔에서 정의 됩니다.  
  
2.  멀티 사이트 배포에 대 한 추가 Active Directory 사이트를 만드는 경우에 추가 사이트에 대 한 새 IPv6 접두사를 계획 하 고 원격 액세스에서 정의 해야 합니다. IPv6 접두사만 구성할 수 있는 IPv6이 회사 내부 네트워크에 배포 하는 경우 원격 액세스 관리 콘솔 또는 PowerShell cmdlet을 사용 하 여 note 합니다.  
  
### <a name="ipv6-prefix-for-directaccess-client-computers-ip-https-prefix"></a>DirectAccess 클라이언트 컴퓨터 (IP HTTPS 접두사)에 대 한 IPv6 접두사  
  
1.  IPv6 내부 회사 네트워크에 배포 하는 경우 배포에 추가 진입점에서 DirectAccess 클라이언트 컴퓨터에 할당할 IPv6 접두사를 계획 해야 합니다.  
  
2.  각 진입점에서 DirectAccess 클라이언트 컴퓨터에 할당할 IPv6 접두사는 고유 하 고 IPv6 접두사에 중복 되는 확인 합니다.  
  
3.  IPv6를 회사 네트워크에 배포 되지 않은 경우 진입점을 추가 하는 경우 각 진입점에 대 한 IP-HTTPS 접두사를 자동으로 선택 됩니다.  
  
### <a name="ipv6-prefix-for-vpn-clients"></a>VPN 클라이언트에 대 한 IPv6 접두사  
VPN 단일 원격 액세스 서버에 배포한 경우 다음 note:  
  
1.  IPv6 VPN을 추가 접두사를 진입점에는 VPN 클라이언트가 회사 네트워크에 IPv6 연결을 허용 하려는 경우 필요 합니다.  
  
2.  VPN 접두사도 IPv6가 내부 회사 네트워크에 배포 하는 경우 및 진입점에서 VPN을 사용 하는 경우 원격 액세스 관리 콘솔 또는 PowerShell cmdlet를 사용 하 여 진입점에만 구성할 수 있습니다.  
  
3.  VPN 접두사도 각 진입점에서 고유 해야 하며 다른 VPN 또는 IP-HTTPS 접두사를 사용 하 여 겹치지 않아야 합니다.  
  
4.  IPv6를 회사 네트워크에 배포 되지 않은 진입점에 연결 하는 VPN 클라이언트는 IPv6 주소를 할당 되지 않습니다.  
  
### <a name="routing"></a>라우팅  
멀티 사이트 배포에서 대칭 라우팅 적용 됩니다 Teredo 및 IP-HTTPS를 사용 하 여. IPv6이 회사 네트워크에 배포 하는 경우 다음 note:  
  
1.  각 진입점의 Teredo 및 IP-HTTPS 접두사는 회사 네트워크 연결 된 해당 원격 액세스 서버를 통해 라우팅할 수 있어야 합니다.  
  
2.  경로 회사 네트워크 라우팅 인프라에서 구성 되어야 합니다.  
  
3.  각 진입점에 대 한 내부 네트워크에 1 ~ 3 개의 경로 수 해야 합니다.  
  
    1.  IP-HTTPS 접두사를이 접두사는 관리자가 선택한 추가 진입점 마법사.  
  
    2.  VPN IPv6 접두사 (선택 사항)입니다. 진입점에 대 한 VPN을 사용 하도록 설정한 후이 접두사를 선택할 수 있습니다.  
  
    3.  Teredo 접두사 (선택 사항)입니다. 이 접두사는 원격 액세스 서버는 두 개의 연속 된 공용 IPv4 주소가 외부 어댑터에 구성 된 경우에 적합 합니다. 접두사 주소 쌍의 첫 번째 공용 IPv4 주소를 기반으로 합니다. 예를 들어 외부 주소:  
  
        1.  www.xxx.yyy.zzz  
  
        2.  www.xxx.yyy.zzz+1  
  
        구성 하려면 Teredo prefix 2001:0:WWXX:YYZZ 됩니다:: / 64 WWXX:YYZZ IPv4 주소 www.xxx.yyy.zzz의 16 진수 표현입니다.  
  
        Teredo prefix를 계산 하려면 다음 스크립트를 사용할 수 있는 note:  
  
        ```  
        $TeredoIPv4 = (Get-NetTeredoConfiguration).ServerName # Use for a Remote Access server that is already configured  
        $TeredoIPv4 = "20.0.0.1" # Use for an IPv4 address  
  
            [Byte[]] $TeredoServerAddressBytes = `  
            [System.Net.IPAddress]::Parse("2001::").GetAddressBytes()[0..3] + `  
            [System.Net.IPAddress]::Parse($TeredoIPv4).GetAddressBytes() + `  
            [System.Net.IPAddress]::Parse("::").GetAddressBytes()[0..7]  
  
        Write-Host "The server's Teredo prefix is $([System.Net.IPAddress]$TeredoServerAddressBytes)/64"  
        ```  
  
    4.  위의 경로의 모든 원격 액세스 서버의 내부 어댑터에 IPv6 주소를 라우팅해야 (또는 분산 된 진입점을 내부 가상 IP (VIP) 주소 부하에 대해).  
  
> [!NOTE]  
> DirectAccess, Teredo에 대 한 경로 통해 원격으로 수행 됩니다 IPv6이 회사 네트워크 및 원격 액세스 서버 관리에 배포 하는 경우 및 트래픽을 수 있도록 각 원격 액세스 서버에 다른 모든 진입점에서 IP-HTTPS 접두사를 추가 해야 합니다. 내부 네트워크로 전달 합니다.  
  
### <a name="active-directory-site-specific-ipv6-prefixes"></a>Active Directory 사이트별 IPv6 접두사  
Windows 10 또는 Windows 8을 실행 하는 클라이언트 컴퓨터는 진입점에 연결할 때 즉시 클라이언트 컴퓨터는 진입점의 Active Directory 사이트에 연결 된 및 진입점와 연결 된 IPv6 접두사를 사용 하 여 구성 됩니다. 기본 설정이 구성 된 동적 IPv6 접두사 정책 테이블에서 우선 순위가 높은 진입점에 연결할 때 하므로 이러한 IPv6 접두사를 사용 하 여 리소스에 연결 하기 위해 클라이언트 컴퓨터입니다.  
  
조직 (예를 들어는 내부 리소스 FQDN app.corp.com에 호스팅되는 North America 및 Europe 모두 각 위치에서 사이트별 IP 주소를 사용 하 여) 사이트별 IPv6 접두사를 사용 하 여 Active Directory 토폴로지를 사용 하는 경우 구성 되지 않았습니다. 각 진입점에 대 한 기본 원격 액세스 콘솔 및 사이트별 IPv6 접두사를 사용 하 여 구성 되지 않습니다. 이 선택적 시나리오를 사용 하도록 설정 하려면, 특정 진입점에 연결 하는 클라이언트 컴퓨터에서 선호 하는 특정 IPv6 접두사를 사용 하 여 각 진입점을 구성 해야 합니다. 다음과 같이이 수행 합니다.  
  
1.  Windows 10 또는 Windows 8 클라이언트 컴퓨터에 사용 되는 각 GPO에 대해 집합 DAEntryPointTableItem PowerShell cmdlet을 실행  
  
2.  사이트별 IPv6 접두사를 사용 하 여 cmdlet에 대 한 EntryPointRange 매개 변수를 설정 합니다. 예를 들어, 사이트별를 추가할 2001:db8:1:1 접두사:: / 64 및 2001:db:1:2:: 다음을 실행 하는 유럽, 호출 하는 진입점 64 /  
  
    ```  
    $entryPointName = "Europe"  
    $prefixesToAdd = @("2001:db8:1:1::/64", "2001:db8:1:2::/64")  
    $clientGpos = (Get-DAClient).GpoName  
    $clientGpos | % { Get-DAEntryPointTableItem -EntryPointName $entryPointName -PolicyStore $_ | %{ Set-DAEntryPointTableItem -PolicyStore $_.PolicyStore -EntryPointName $_.EntryPointName -EntryPointRange ($_.EntryPointRange) + $prefixesToAdd}}  
    ```  
  
3.  EntryPointRange 매개 변수를 수정할 때는 IPsec 터널 끝점 및 DNS64 주소에 속하는 기존 128 비트 접두사를 제거 하지 않는 것을 확인 합니다.  
  
## <a name="bkmk_3_7_TransitionIPv6"></a>3.7 멀티 사이트 원격 액세스를 배포할 때 IPv6로 전환 계획  
대부분의 조직에서는 회사 네트워크의 IPv4 프로토콜을 사용 합니다. 사용 가능한 IPv4 접두사의 소모를 사용 하 여 대부분의 조직에서는 IPv6 전용 네트워크에서 IPv4 전용로 전환을 하는 합니다.  
  
이 상태로 전환 될 가능성이 가장 높은 두 단계로 수행 합니다.  
  
1.  IPv4 전용에서 IPv6 + IPv4 회사 네트워크입니다.  
  
2.  IPv6 + IPv4에서 IPv6 전용 회사 네트워크입니다.  
  
각 부분에서는 전환 단계에서 수행할 수 있습니다. 각 단계에서 네트워크의 서브넷이 하나만 새 네트워크 구성을 변경할 수 있습니다. 따라서 DirectAccess 멀티 사이트 배포는 where, 예를 들어, 진입점의 일부는 IPv4 전용 서브넷에 속하고 IPv6 + IPv4 서브넷에 속하는 다른 하이브리드 배포를 지원 해야 합니다. 또한 변환 프로세스 중에 구성 변경 DirectAccess 통한 클라이언트 연결을 중단 하지 해야 합니다.  
  
### <a name="TransitionIPv4toMixed"></a>IPv6 + IPv4 회사 네트워크에 IPv4 전용에서 전환  
IPv6 주소는 IPv4 전용 회사 네트워크에 추가할 때에 이미 배포 된 DirectAccess 서버에 IPv6 주소를 추가 하는 것이 좋습니다. 또한 다음 DirectAccess 배포에 IPv4와 IPv6 주소를 사용 하 여 부하 분산 된 클러스터에 진입점 또는 노드를 추가 하는 것이 좋습니다.  
  
원격 액세스를 사용 하면 원래 IPv4 주소만 사용 하 여 구성 된 배포에 IPv4와 IPv6 주소를 사용 하 여 서버를 추가할 수 있습니다. 이러한 서버는 IPv4 전용 서버로 추가 되 고 해당 IPv6 주소는 DirectAccess;에서 무시 됩니다. 결과적으로 조직에서 이러한 새 서버의 기본 IPv6 연결의 장점 활용을 사용할 수 없습니다.  
  
IPv6 + IPv4 배포 하도록 배포를 변환 하 고 네이티브 IPv6 기능을 활용 하려면 DirectAccess을 다시 설치 해야 합니다. 유지 관리를 설치 하는 전체 클라이언트 연결 IPv4 전용에서 전환 이중 DirectAccess 배포를 사용 하 여 IPv6 전용 배포에 참조 합니다.  
  
> [!NOTE]  
> 혼합 IPv4와 IPv6 네트워크에서는 IPv4 전용 네트워크와 마찬가지로 클라이언트 DNS 요청을 확인 하는 데 사용 되는 DNS 서버의 주소를 회사 DNS와 하지 자체에 대 한 원격 액세스 서버에 배포 된 DNS64를 사용 하 여 구성 되어야 합니다.  
  
### <a name="TransitionMixedtoIPv6"></a>IPv6 전용 회사 네트워크에서 IPv6 + IPv4 전환  
DirectAccess를 사용 하면 IPv6 전용 진입점 경우 배포의 첫 번째 원격 액세스 서버에 있던 하거나 IPv4 및 IPv6 주소 이거나 IPv6 주소일만 추가할 수 있습니다. 즉, DirectAccess를 다시 설치 하지 않고 단일 단계에서 IPv6 전용 네트워크로 IPv4 전용 네트워크에서 전환할 수 없습니다. 있습니다. IPv6 전용 네트워크는 IPv4 전용 네트워크에서 직접 전환, 전환 IPv4 전용에서 이중 DirectAccess 배포를 사용 하 여 IPv6 전용 배포로 참조 하세요.  
  
IPv4 전용 배포로 전환 IPv6 + IPv4 배포를 완료 한 후에 IPv6 전용 네트워크로 전환할 수 있습니다. 하는 동안 전환 된 후, 다음 note:  
  
-   회사 네트워크의 IPv4 전용 백 엔드 서버 유지 하는 경우 IPv6 전용 진입점을 통해 연결 하는 클라이언트에 연결할 수 됩니다.  
  
-   IPv6 전용 진입점에는 IPv4와 IPv6 배포를 추가할 때 DNS64 및 NAT64 됩니다 하지 사용할 새 서버. 회사 DNS 서버를 사용 하도록 이러한 진입점에 연결 된 클라이언트를 자동으로 구성 됩니다.  
  
-   배포 된 서버의 IPv4 주소를 삭제 하는 경우 DirectAccess 배포에서 서버를 제거, 해당 IPv4 회사 네트워크 주소를 제거 하며 배포에 다시 추가 합니다.  
  
회사 네트워크에 클라이언트 연결을 지원 하려면 IPv6 주소를 회사 DNS에서 네트워크 위치 서버를 해결할 수 있음을 확인 해야 합니다. 추가 IPv4 주소도 설정할 수 있지만 필요 하지 않습니다.  
  
### <a name="DualDeployment"></a>이중 DirectAccess 배포를 사용 하 여 IPv6 전용 배포에는 IPv4 전용에서 전환  
IPv6 전용 회사 네트워크에서 IPv4 전용 전환 DirectAccess 배포를 다시 설치 하지 않고 수행할 수 없습니다. 전환 하는 동안 클라이언트 연결을 유지 하려면 다른 DirectAccess 배포를 사용할 수 있습니다. 첫 번째 전환 단계 (IPv4 전용 네트워크 IPv4와 IPv6로 업그레이드)를 완료 하 고 네이티브 IPv6 연결 이점은 이후 전환할은 IPv6 전용 회사 네트워크 셀이나현재 활용에 대 한 준비 하려는 때 이중 배포 필요 합니다. 이중 배포는 다음과 같은 일반적인 단계에 설명 되어 있습니다.  
  
1.  두 번째 DirectAccess 배포를 설치 합니다. 새 서버에서 DirectAccess를 설치할 또는 첫 번째 배포에서 서버를 제거 하 고 두 번째 배포에 사용할 수 있습니다.  
  
    > [!NOTE]  
    > 현재는 함께 추가 DirectAccess 배포를 설치할 때 두 진입점이 없음 같은 클라이언트 접두사를 공유 하는 있는지 확인 합니다.  
    >   
    > DirectAccess 시작 마법사를 사용 하 여 설치 하는 경우 또는 cmdlet `Install-RemoteAccess`, 원격 액세스 배포를 기본 값에서 첫 번째 진입점의 클라이언트 접두사를 자동으로 설정 < IPv6 서브넷\_접두사 >: 1000:: / 64입니다. 필요한 경우에 접두사를 변경 해야 합니다.  
  
2.  첫 번째 배포에서 선택한 클라이언트 보안 그룹을 제거 합니다.  
  
3.  두 번째 배포에는 클라이언트 보안 그룹을 추가 합니다.  
  
    > [!IMPORTANT]  
    > 프로세스 전반에서 클라이언트 연결을 유지 하려면 첫 번째에서 제거 직후 두 번째 배포에 보안 그룹을 추가 해야 합니다. 이렇게 하면 클라이언트를 두 개 또는 0 DirectAccess Gpo를 사용 하 여 업데이트 되지 않습니다. 클라이언트를 검색 하 고 해당 클라이언트 GPO 업데이트 배포를 사용 하 여 시작 됩니다.  
  
4.  선택 사항: 첫 번째 배포에서 DirectAccess 진입점을 제거 하 고 두 번째에서 새 진입점으로 해당 서버를 추가 합니다.  
  
전환 완료 경우 첫 번째 DirectAccess 배포를 제거할 수 있습니다. 를 제거 하는 경우 다음과 같은 문제가 발생할 수 있습니다.  
  
-   배포 모바일 컴퓨터에만 클라이언트를 지원 하도록 구성 된 경우 WMI 필터가 삭제 됩니다. 두 번째 배포의 클라이언트 보안 그룹은 데스크톱 컴퓨터를 포함 하는 경우 DirectAccess 클라이언트 GPO 데스크톱 컴퓨터를 필터링 하지 않습니다 및 에서도 문제가 발생할 수 있습니다. 모바일 컴퓨터 필터를 필요한 경우 다음 지침에 따라 다시 [GPO에 대 한 WMI 필터 만들기](https://technet.microsoft.com/library/cc947846.aspx)합니다.  
  
-   두 배포 모두 원래 동일한 Active Directory 도메인에서 만들어진 경우 DNS 검색 항목 localhost 가리키는 삭제 되 고 클라이언트 연결 문제가 발생할 수 있습니다. 예를 들어 클라이언트 Teredo 보다는 IP-HTTPS를 사용 하 여 연결 또는 DirectAccess 멀티 사이트 진입점 사이 전환 합니다. 이 경우 회사 DNS에는 다음과 같은 DNS 항목을 추가 해야 합니다.  
  
    -   영역: 도메인 이름  
  
    -   이름: directaccess corpConnectivityHost  
  
    -   IP 주소::: 1  
  
    -   형식: AAAA
